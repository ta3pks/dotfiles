    - Decision: We will continue using manual symlinks rather than adopting GNU Stow.

    - Context: The XDG Base Directory Specification defines standard locations for user configs, data, cache, and state. Many tools support it, but not all.
    - Decision: We will organize configurations following the XDG Base Directory Specification where tools support it.
    - Status: Accepted
    - Consequences:
      - Positive: Cleaner home directory, consistent structure
      - Positive: Easier to backup/restore configs
      - Negative: Some tools require non-XDG locations (e.g., .zshrc)
      - Neutral: Must respect $XDG_* environment variables with fallbacks

    ```markdown
    # Dotfiles Philosophy

    Design principles guiding this repository.

    ## Manual Symlinks Over Stow

    We use explicit manual symlinks rather than GNU Stow. This provides:
    - No external dependency for core functionality
    - Full control over symlink behavior
    - Easy debugging when links break
    - Support for custom features (--dry-run, --undo, per-file prompts)

    See [ADR-001](decisions/001-manual-symlinks-over-stow.md).

    ## XDG Compliance

    Configurations follow the XDG Base Directory Specification where supported:
    - Configs: `$XDG_CONFIG_HOME` (default: ~/.config)
    - Data: `$XDG_DATA_HOME` (default: ~/.local/share)
    - Cache: `$XDG_CACHE_HOME` (default: ~/.cache)

    This keeps the home directory clean and makes backup/restore straightforward.

    See [ADR-002](decisions/002-xdg-base-directory.md).

    ## Tool Selection Criteria

    Tools are selected based on:
    1. **Native Wayland support** (sway over i3, waybar over polybar)
    2. **Active maintenance** (avoid abandoned projects)
    3. **Lua configuration** (nvim, waybar) where possible
    4. **Minimal dependencies** (fewer moving parts = fewer failures)

    ## Documentation for Future Self

    All documentation assumes the reader (future me) has forgotten everything.
    - Document WHY, not just WHAT
    - ADRs for architectural decisions
    - Inline comments for small choices
    - Per-tool READMEs for configuration details

    ## No Over-Engineering

    Start simple, add complexity as needed:
    - No CI/CD for dotfiles (manual verification sufficient)
    - No cross-platform support (Linux only, macOS configs separate)
    - No templating system (static configs work fine)
    ```
