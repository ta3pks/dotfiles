# Plan 07-01: Export/Import Functionality

## Goal
Enable users to export memories to JSON files and import them back, supporting data portability and migration scenarios.

## Context
- Uses existing memory service APIs: `listAllMemories()`, `storeMemory()`, `removeMemory()`
- CLI pattern: `register*Command` function with option chain
- Dependencies already in project: commander, inquirer, chalk, zod, ora
- Wave 0 helper: `tests/cli/helpers/setup.ts` provides shared test fixtures

## Tasks

### Task 1: Implement Export Command
```xml
<task type="auto">
  <name>Implement Export Command</name>
  
  <files>
    memory/src/cli/commands/export.ts

  <action>
```

### Task 2: Implement Import Command
```xml
<task type="auto">
  <name>Implement Import Command</name>
  
  <files>
    memory/src/cli/commands/import.ts
    memory/src/cli/index.ts
    tests/cli/import.test.ts
  </files>
  
  <action>
    1. Create registerImportCommand(program) following existing CLI patterns
    2. Implement options:
       - --input, -i: input directory path (required)
       - --merge: add to existing memories (default mode)
       - --replace: delete all memories before import
       - --on-duplicate: skip | update | prompt (default: skip)
       - --yes: skip confirmation prompt
    3. Create Zod schema for memory file validation:
       - Required: content (string)
       - Optional: tags (array), project (string|null), createdAt (ISO date)
    4. Handler logic:
       - If --replace, prompt for confirmation (unless --yes)
       - If --replace confirmed, remove all existing memories
       - Read all JSON files from input directory
       - Validate each file against Zod schema
       - For valid files: call storeMemory() with content, tags, project
       - For duplicates: handle per --on-duplicate option
       - Track: imported, skipped, updated, failed counts
       - Report summary at end
    5. Error handling:
       - CLIError if input path doesn't exist
       - Continue on validation failures, report at end
       - Prompt mode uses inquirer for each duplicate
    6. Register command in cli/index.ts
    7. Create tests/cli/import.test.ts with tests for: memory creation, --merge, --replace, --on-duplicate modes, validation, roundtrip
    8. Use test helper from tests/cli/helpers/setup.ts for fixtures
  </action>
  
  <verify>
    <automated>bun test tests/cli/import.test.ts</automated>
    <manual>Export memories, delete all, run: memory import --input ./exports --replace --yes, verify roundtrip preserves data</manual>
  </verify>
  
  <done>
    - memory import command reads JSON files and stores memories
    - --merge mode adds to existing memories
    - --replace mode clears all memories before import
    - --on-duplicate handles conflicts (skip/update/prompt)
    - Zod schema validates JSON structure
    - All import tests pass
  </done>
</task>
```

## Dependencies
- Phase 5: Memory storage service (storeMemory, listAllMemories)
- Phase 6: Memory retrieval service (removeMemory for --replace mode)
- Plan 07-00: Test helper setup (tests/cli/helpers/setup.ts)

## Risks
- Large exports may be slow; consider progress indicator with ora
- Import validation strictness may reject valid user data; balance strictness with usability
- Duplicate detection requires comparing content or generating stable IDs
