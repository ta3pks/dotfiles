# Plan 07-02: Backup/Restore and Cleanup Functionality

## Goal
Enable users to create and restore database backups, and clean up old or irrelevant memories to maintain a healthy memory database.

## Context
- Backup: Copy SQLite database file to timestamped location
- Restore: Replace current database with backup file
- Cleanup: Use `removeMemory()` to delete memories matching criteria
- Dependencies already in project: commander, inquirer, chalk, ora
- Wave 0 helper: `tests/cli/helpers/setup.ts` provides shared test fixtures

## Tasks

### Task 1: Implement Backup and Restore Commands
```xml
<task type="auto">
  <name>Implement Backup and Restore Commands</name>
  
  <files>
    memory/src/cli/commands/backup.ts
    memory/src/cli/index.ts
    tests/cli/backup.test.ts
  </files>
  
  <action>
    1. Create registerBackupCommand(program) following existing CLI patterns
    2. Backup options:
       - --output, -o: backup directory (default: ~/.local/share/memory/backups/)
       - --max-backups: keep only N most recent backups (default: 10)
       - --name: custom backup name (overrides timestamp)
    3. Backup handler logic:
       - Create backup directory if not exists
       - Generate timestamped filename: memory-backup-YYYY-MM-DDTHH-mm-ss.db
       - Copy current database file using fs.copyFile
       - If --max-backups set, list existing backups and delete oldest beyond limit
       - Report: backup path, size, rotation summary
    4. Restore as subcommand: memory restore --file <path>
       - --file, -f: backup file path (required)
       - --list: list available backups (no restore)
       - --yes: skip confirmation prompt
    5. Restore handler logic:
       - If --list, show available backups with dates and sizes
       - Validate backup file exists and is valid SQLite
       - Prompt for confirmation (unless --yes)
       - Create safety backup of current database before restore
       - Copy backup file to database location
       - Report: restore source, safety backup location
    6. Error handling:
       - CLIError if database/backup file doesn't exist
       - CLIError if backup directory not writable
       - CLIError if backup file is not valid SQLite
    7. Register both commands in cli/index.ts
    8. Create tests/cli/backup.test.ts with tests for: timestamped backup, --output, --max-backups rotation, restore, --list, validation
    9. Use test helper from tests/cli/helpers/setup.ts for fixtures
  </action>
  
  <verify>
    <automated>bun test tests/cli/backup.test.ts</automated>
    <manual>Run: memory backup && memory restore --list && verify timestamped .db file exists</manual>
  </verify>
  
  <done>
    - memory backup creates timestamped .db file in default or custom location
    - --max-backups rotates old backups, keeping only N most recent
    - memory restore reverts database to backup state
    - restore --list shows available backups with dates and sizes
    - restore validates backup file before overwriting
    - All backup/restore tests pass
  </done>
</task>
```

### Task 2: Implement Cleanup Command
```xml
<task type="auto">
  <name>Implement Cleanup Command</name>
  
  <files>
    memory/src/cli/commands/cleanup.ts
    memory/src/cli/index.ts
    tests/cli/cleanup.test.ts
  </files>
  
  <action>
    1. Create registerCleanupCommand(program) following existing CLI patterns
    2. Implement options:
       - --older-than: age threshold (e.g., 30d, 6m, 1y)
       - --tag: filter by tag
       - --project: filter by project
       - --type: filter by memory type (if applicable)
       - --dry-run: show what would be deleted without deleting
       - --yes: skip confirmation prompt
    3. Age parsing:
       - Support formats: Nd (days), Nw (weeks), Nm (months), Ny (years)
       - Convert to Date for comparison with memory.createdAt
    4. Handler logic:
       - Initialize memory service
       - Call listAllMemories() to get candidates
       - Apply filters (age, tag, project, type)
       - If --dry-run: show list with details, exit
       - If not --yes: show summary, prompt for confirmation
       - For each matching memory: call removeMemory(id)
       - Report: deleted count, freed space estimate
    5. Error handling:
       - CLIError if invalid --older-than format
       - Continue on individual removal failures, report at end
    6. Register command in cli/index.ts
    7. Create tests/cli/cleanup.test.ts with tests for: --older-than, --dry-run, --tag, --project, --yes, multi-criteria
    8. Use test helper from tests/cli/helpers/setup.ts for fixtures
  </action>
  
  <verify>
    <automated>bun test tests/cli/cleanup.test.ts</automated>
    <manual>Run: memory cleanup --older-than 30d --dry-run && verify shows only memories older than 30 days</manual>
  </verify>
  
  <done>
    - memory cleanup --older-than filters and deletes memories by age
    - --dry-run shows what would be deleted without actually deleting
    - --tag and --project filter by tag and project
    - --yes skips confirmation prompt
    - Multi-criteria filtering combines age, tag, project, type
    - All cleanup tests pass
  </done>
</task>
```

## Dependencies
- Phase 5: Memory storage service (listAllMemories)
- Phase 6: Memory removal service (removeMemory)
- SQLite database location from config
- Plan 07-00: Test helper setup (tests/cli/helpers/setup.ts)

## Risks
- Backup during active writes may produce inconsistent backup; recommend quiet period
- Restore overwrites current database; always create safety backup first
- Cleanup with broad criteria may delete important data; dry-run by default

## Success Criteria Met
- [x] User can create and restore backups of the memory database
- [x] User can prune old or irrelevant memories to keep database clean
